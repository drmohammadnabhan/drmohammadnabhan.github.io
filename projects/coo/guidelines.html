Practitioner Guidelines

When and How to Use COO

When to Use COO

COO is well-suited for constrained nonlinear optimization problems with the following characteristics:

Smooth objective and constraints. \(f\) and \(G_i\) are continuously differentiable, and gradients are available — analytically, via automatic differentiation, or via finite differences. The Störmer-Verlet integrator requires Lipschitz-continuous gradients (Assumption A1).

Active constraints at the optimum. The engineering design has binding inequality constraints — walls that the optimal solution presses against. COO’s geometric insight is most powerful at constraint boundaries. If the optimal solution is in the interior of the feasible set, standard unconstrained Nesterov acceleration (NAG/FISTA) is sufficient and simpler.

Shadow prices needed. You want to know not just what the optimal design is, but why it cannot be better — which constraint is limiting, and by how much. COO generates KKT multiplier estimates throughout the optimization run, not only at the terminal point.

Moderate-to-high dimension with expensive gradients. COO’s \(O(\varepsilon^{-1/3})\) primal rate and flat ~1,500 gradient evaluation budget make it competitive for simulation-based design optimization where each gradient call involves a finite-element solve, CFD run, or experimental measurement.

Multiple active constraints (5–20+). COO handles each constraint boundary independently via sequential Snell-KKT reflections. The 11-constraint speed reducer (P5) is the benchmark showing this most clearly — where penalty gradient descent fails entirely.

When NOT to Use COO (Current Version)

Narrow feasible corridors. If the feasible region is a thin tunnel (e.g., the welded beam or spring problems), sequential reflections trap the trajectory between opposing walls. Standard gradient projection or interior-point methods are preferable for such geometry.

Equality constraints. The current algorithm handles inequality constraints. Equality constraints can be split into two inequalities (\(h(x) = 0 \Rightarrow h(x) \leq 0\) and \(-h(x) \leq 0\)), but this doubles the constraint count and may degrade performance.

Non-smooth objectives. The leapfrog integrator requires \(L\)-smooth \(f\). For non-smooth problems, use subgradient methods or smoothing.

Global optimality required. COO is a local optimizer — Theorem 4 guarantees an ε-KKT point, not the global minimum. For guaranteed global solutions on non-convex problems, combine COO with a global search strategy: use PSO or a grid search for global exploration, COO for local refinement with shadow prices.

Parameters

Parameter Default Role Tuning Guidance
\(s = 1/L\) Auto-estimated Lumen step size Estimated from initial gradient difference; rarely needs adjustment. If the Lumen trajectory shows energy drift, reduce by 50%.
\(h\) \(0.1/L\) Ombra step size Keep \(hL < 1\) for stability. Smaller = more stable, larger = faster per Ombra step.
\(\gamma\) \(0.5\) Ombra damping Higher = faster local convergence. If Ombra overshoots near flat minima, reduce to 0.2.
\(\tau\) \(10\) APF percentile threshold Raise to 20–30 if Ombra fires too rarely (too few \(x_{\text{best}}\) updates). Lower to 5 if Ombra fires every few steps but doesn’t improve \(f_{\text{best}}\).
\(C\) \(50\) APF cooldown (steps) Prevents repeated Ombra spawns from the same trajectory region. Increase for high-dimensional problems.
\(N_{\max}\) \(5000\) Max Lumen iterations The primary budget parameter. Increase for problems with many constraints or large feasible regions.
\(N_{\text{Ombra}}\) \(300\) Max Ombra steps Sufficient for most problems under the exponential convergence rate. Increase only if Ombra regularly hits this limit without converging.

Recommended starting configuration

For a new problem, begin with all defaults. Run once and inspect:

  1. How many times did the Ombra fire? (check k_last history) — If fewer than 3 times, lower \(\tau\).
  2. Did \(f_{\text{best}}\) improve after each Ombra? — If not, check that the spawn point is feasible and that \(H_0\) at spawn is indeed low.
  3. Has the Luminosity Flux \(\bar\Lambda_k\) stabilized? — Plot it. If still varying at termination, increase \(N_{\max}\).

Implementation Checklist

Before running COO on a new problem:

Reading the Shadow Prices

The Luminosity Flux output \(\bar{\Lambda} = [\bar\lambda_1, \ldots, \bar\lambda_m]\) contains the averaged KKT multiplier for each constraint.

Interpretation:

  • \(\bar\lambda_i \approx 0\): constraint \(i\) is slack at the optimum — relaxing it would not improve the objective.
  • \(\bar\lambda_i > 0\): constraint \(i\) is binding — costing you \(\bar\lambda_i\) units of objective improvement per unit of constraint relaxation.
  • The constraint with the largest \(\bar\lambda_i\) is the binding bottleneck — the first place to invest redesign effort.

Convergence check. Plot \(\bar\Lambda_k\) over iterations. It should stabilize to a constant vector as the algorithm progresses (Theorem 5: \(O(1/k^2)\) convergence). If it has not stabilized at termination, increase \(N_{\max}\). Instability after 3,000+ steps usually indicates the trajectory has not accumulated enough boundary events near \(x^*\) — check whether the Lumen is spending time near the active constraints.

Note on individual events. Individual multiplier values \(\lambda_i^{(j)}\) at each boundary event decay to zero as the algorithm converges (the approach momentum decreases as the orbit tightens). The average \(\bar\Lambda_k\) correctly recovers \(\lambda^*\) even as individual events become small — this is by design, not a numerical artifact. See the Methodology page for the formal result.
